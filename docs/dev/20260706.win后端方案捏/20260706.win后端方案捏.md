# Windows 后端方案草案

日期：2026-07-06

## 目标

- Windows 默认从 Rust 侧采集硬件信息。
- 默认不启动 `powershell.exe` / `pwsh.exe`。
- 只有显式传 `--powershell` 时才走现有 PowerShell 后端。
- 输出尽量接近当前 `hdrt --powershell`。
- 增强后端失败时退回现有 `sysinfo + registry` basic 后端。

## 当前状态

默认 Windows backend 在 `src/collector/windows/basic.rs`：

- 磁盘：registry `SYSTEM\CurrentControlSet\Services\disk\Enum`
- 内存：`sysinfo` 只能拿系统总量
- CPU：`sysinfo + registry`
- 主板 / BIOS：registry `HARDWARE\DESCRIPTION\System\BIOS`

PowerShell backend 在 `src/collector/windows/powershell.rs`：

- 脚本：`src/collector/windows/scripts/collect_hardware.ps1`
- 数据源：`Get-PhysicalDisk` + `Get-CimInstance Win32_*`

默认模式很多 `Unknown` 的原因：registry 的磁盘枚举只适合基础 PnP inventory，不是完整 CIM / Storage inventory，所以 size、health、真实 serial、firmware 等字段经常拿不到。

## 路线排序

按运行性能和系统开销，从快到慢大致是：

1. 底层 WinAPI / DeviceIoControl / SMBIOS
2. 手写 WMI COM
3. Rust `wmi` crate
4. PowerShell backend

判断：

- 第 1 条理论最快，但完整覆盖最难。
- 第 2、3 条性能接近，瓶颈主要是 WMI 服务本身。
- 第 3 条开发最快，适合先落地默认增强后端。
- 第 4 条最慢，但已经可用，适合作为显式对照和 debug fallback。

## Phase 1：Native WMI 默认增强

目标：

- 新增 `native_wmi` backend。
- 默认 Windows 采集先尝试 `native_wmi`。
- `native_wmi` 失败或返回空时退回 `basic`。
- `--powershell` 行为保持不变，只在显式传参时使用。

数据源：

- `ROOT\Microsoft\Windows\Storage: MSFT_PhysicalDisk`
- `ROOT\CIMV2: Win32_DiskDrive`
- `ROOT\CIMV2: Win32_PhysicalMemory`
- `ROOT\CIMV2: Win32_Processor`
- `ROOT\CIMV2: Win32_BaseBoard`
- `ROOT\CIMV2: Win32_BIOS`

实现建议：

- 先用 Rust `wmi` crate。
- 映射到现有 `DiskInfo`、`MemoryDevice`、`CpuInfo`、`MotherboardInfo`。
- 磁盘优先 `MSFT_PhysicalDisk`，失败或为空再用 `Win32_DiskDrive`。
- WMI 报错时不要 panic，生成 warning 后 fallback。

预期收益：

- 默认不启动 PowerShell。
- 内存条 slot、manufacturer、part number、serial 大概率能拿到。
- 磁盘 size、bus、firmware、health 比 registry 完整。
- 主板 / BIOS 字段更接近 `--powershell`。

验收命令：

- `hdrt`
- `hdrt disk`
- `hdrt memory`
- `hdrt motherboard`
- `hdrt --powershell`
- `hdrt --format json`

必须确认：

- 不传 `--powershell` 时不会启动 PowerShell。
- WMI 失败时仍能输出 basic report。
- warning 能说明 fallback 原因。

## Phase 2：Benchmark / Doctor 对照

目标：增加耗时对照入口，真实比较 backend 成本。

候选命令：

- `hdrt doctor --bench`
- 或 `hdrt bench`

第一批后端：`basic`、`native_wmi`、`powershell`。

输出字段：backend 名称、是否成功、耗时、disks / memory rows 数量、warning / error 摘要。

示例：

```text
backend       ok   elapsed   disks   memory   note
basic         yes  35ms      14      1        sysinfo + registry
native-wmi    yes  180ms     7       2        WMI COM
powershell    yes  950ms     7       2        external powershell.exe
```

## Phase 3：WinAPI Storage 补强

目标：

- 先不做完整 SMART。
- 只做 `IOCTL_STORAGE_QUERY_PROPERTY`。
- 用它补磁盘 vendor、product、firmware、serial、bus。

参考：

- OpenHardwareMonitor `WindowsStorage.cs`
- `CreateFileW("\\\\.\\PhysicalDriveN")`
- `DeviceIoControl(IOCTL_STORAGE_QUERY_PROPERTY)`
- `STORAGE_DEVICE_DESCRIPTOR`

集成方式：

- `native_wmi` 负责磁盘列表、size、health。
- `winapi_storage` 用 drive index 补 descriptor 字段。
- 不把它做成第一版唯一磁盘来源。

风险：某些设备需要管理员权限或更宽松 share/access；USB / RAID / 虚拟盘字段可能不稳定；需要小心 `\\.\PhysicalDriveN` 和 WMI disk index 的映射。

## Phase 4：SMBIOS 解析

目标：用 `GetSystemFirmwareTable(RSMB)` 解析 SMBIOS，补强 memory、baseboard、BIOS、CPU socket/cache 等字段。

参考：

- OpenHardwareMonitor `FirmwareTable.cs`
- OpenHardwareMonitor `SMBios.cs`

适合字段：

- BIOS vendor / version / date
- BaseBoard manufacturer / product / version / serial
- Memory locator / bank / manufacturer / part number / serial / speed / size
- CPU socket / core count / thread count / max speed

集成方式：

- 先作为 `native_wmi` 的补充来源。
- 字段冲突时优先使用非 `Unknown` 且更可信的值。

风险：SMBIOS 字段由厂商填写，可能是 `Default string` 或空值；解析代码比 WMI 复杂；需要处理结构长度和版本差异。

## Phase 5：SMART / NVMe Health

目标：

- 在 `--detail smart` 或 `--detail full` 下读取更底层健康信息。
- 默认 basic/detail 不做重型探测。

ATA / SATA 路线：

- `DeviceIoControl`
- `DFP_SEND_DRIVE_COMMAND`
- `DFP_RECEIVE_DRIVE_DATA`
- `SMART_READ_DATA`
- `RETURN_SMART_STATUS`

NVMe 路线：

- `IOCTL_STORAGE_QUERY_PROPERTY`
- `STORAGE_PROTOCOL_SPECIFIC_DATA`
- Identify Controller
- Health Info Log

参考：

- OpenHardwareMonitor `WindowsSmart.cs`
- OpenHardwareMonitor `NVMeWindows.cs`

风险：设备覆盖复杂；NVMe 厂商、Intel RST、Samsung、USB bridge 行为可能不同；权限和驱动差异会导致失败；不适合一开始进入默认路径。

## 默认决策链

建议最终 Windows 默认逻辑：

```text
if --powershell:
    powershell_backend()
else:
    native_wmi_backend()
        .enrich_with_winapi_storage_if_available()
        .fallback_to_basic_on_error()
```

后续 `--detail smart/full`：

```text
report = default_windows_backend()

if detail >= smart:
    enrich_with_smart_or_nvme_health(report)
```

## OpenHardwareMonitor 参考价值

有参考价值：

- 磁盘 WMI 枚举和 Storage namespace 关系。
- `IOCTL_STORAGE_QUERY_PROPERTY` descriptor 读取。
- ATA SMART 和 NVMe Health 的 DeviceIoControl 路线。
- SMBIOS 解析结构。

不建议直接照搬：

- C# 实现不能直接用于 Rust。
- 项目是 MPL-2.0，需要注意许可证边界。
- WinRing0 / InpOut 驱动级访问不适合 hdrt 默认路径。
- 它偏传感器监控，hdrt 当前更像硬件 inventory CLI/TUI。

## 第一轮最小可交付

第一轮做：

- 新增 `src/collector/windows/native_wmi.rs`。
- 新增 Windows 依赖 `wmi`。
- 默认 Windows backend 优先调用 native WMI。
- `--powershell` 保持显式 opt-in。
- fallback 到现有 `basic`。
- 更新 `doctor` 文案，说明 native WMI / PowerShell / basic 的关系。

第一轮不做：手写 COM、SMBIOS parser、SMART / NVMe health、WinRing0 / 驱动级访问。

成功标准：

- `hdrt` 默认输出比现在少很多 `Unknown`。
- `hdrt` 默认不会启动 PowerShell。
- `hdrt --powershell` 仍然可用于对照。
- 失败路径不会 panic。
- JSON / table / markdown 输出结构不变。
- Windows 非管理员用户也能尽量输出信息，并给出清晰 warning。
